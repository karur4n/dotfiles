- プランを提示するときは over-engineered / under-engineered / perfectly-engineered のどれかを判断し、理由を述べる
- 英語ジャーゴンを使わない

## ルール

### 文章・ドキュメント

- 簡潔第一・肯定形で書く。詳しさは求められてから足す
- 設計ドキュメントは仕様（あるべき姿）のみ。実装ステータス・段取り・セッション経過を混ぜない
- 独自の略号を発明しない。実名（テーブル名・関数名）で平易な日本語で書く
- DB スキーマは最終形の DDL（CREATE TABLE）で示し、差分は説明文で述べる

### コード

- コメントは最小限・英語。非自明な知見だけ `// NOTE:` で残す
- 暗黙のフォールバックやリトライより即エラーで落とす
- 実装の同語反復になるテストは書かない

### 設計

- 最小集合から始め、必要になった時点で足す
- 修正説明で他レイヤーの挙動が主語になったら、そちらが真の修正位置か疑う

### 調査・報告

- 確認済みの事実と推測を分ける。データの断定は DB/BigQuery への1クエリで確定させてから
- コマンドの空出力を根拠にしない。先に終了コードと stderr を確認する

### 進め方

- 実装着手前に確認する: 既存 PR/スレッド続報・マージ先ブランチ・（決済なら）対象チャネル
- 音声入力由来らしき不自然な語は推測せず1問確認してから着手する
- 指示された範囲だけ変更する。関連修正・追加提案は確認してから。PR に無関係な diff を混ぜない
- PR 作成後の origin への push 前は確認を取る
- 調査結果は repo 内 docs が既定。Notion 起票は明示依頼時のみ
