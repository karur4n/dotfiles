## 作業スタイル（過去の会話で繰り返し指摘された好み）

### 文章・ドキュメント

- 簡潔第一。PR description・説明・ドキュメントは最小の文章量で書く。詳しさは求められてから足す
- 肯定形で書く。「持たないもの」「しないこと」の列挙はノイズ
- 設計ドキュメントは仕様（あるべき姿）のみ。実装ステータス（進行中・マージ済・Phase X）や段取りメモを混ぜない
- セッションの経過・議論の履歴を成果物に残さない。レビュアーが読むのは最終形だけ
- 独自の略号・記号表記を発明しない（例: `R=read / INS=insert`）。代名詞ではなく実名（テーブル名・関数名）で書く
- 平易な日本語で書く。コード識別子と正式なエンティティ名はそのまま

### コード

- コメントは最小限・英語。説明的コメントは書かず、残す価値のある非自明な知見だけ `// NOTE:` で残す
- 暗黙のフォールバック（`?? 0` 等）や防御的なリトライより、即エラーで落とす方を好む
- 実装の同語反復になるテストは書かない。簡単な変更（定数追加・map キー追加・単純分岐）にテスト追加は不要

### 設計

- 最小集合から始める。enum 値・列・抽象化を将来用に広げない。必要になった時点で足す
- 修正位置はレイヤーをまたいで検討する。修正説明で他レイヤーの挙動を主語にし始めたら、そちらが真の修正位置の可能性が高い

### 進め方

- 指示された範囲だけを変更する。補足・関連修正・追加提案は確認してから
- PR の diff に無関係なファイルを混ぜない。混入したら rebase で取り除く
- PR 作成後に origin へ push する前は確認を取る。ローカルの編集・コミット・テストは進めてよい
- 調査結果の管理は repo 内 docs が既定。Notion 起票は明示的に依頼されたときだけ
